01

Upload & persist

Multipart file (txt / md / docx / pdf) accepted; temp file written to disk and passed to the extraction service.

routes/ingest.py ingest_upload()
02

Text extraction

PDFs default to Azure OCR (better for scanned Hebrew RTL) with pdfplumber as fallback for native PDFs. DOCX via python-docx. Reversed RTL detection triggers re-escalation to OCR if the initial extraction looks flipped.

services/extraction.py extract_text() Azure OCR
03

Extraction quality gate

Density sanity check: MIN_CHARS_PER_PAGE = 200. Low-yield PDFs are rejected as likely OCR failures rather than silently poisoning the corpus.

routes/ingest.py (lines 65–81)
04

Document row created

A Document row is inserted with metadata: filename, extractor used, used_ocr flag, page count, extraction quality. Flushed to DB to reserve the ID for chunks to reference.

models.Document → document_id
05

Structural chunking

Splits on Hebrew section markers — סעיף, פרק, החלטה, subsections, appendices. Paragraph-based fallback for unstructured text. Sections larger than 3500 characters get re-split. Emits StructuralChunk{text, section_path, position}.

services/chunking.py chunk_document()
06

Contextual embedding input

Before embedding, each chunk gets its document title and section path prepended. E.g. תקנון פנסיה — סעיף 4, then the chunk text. This anchors the embedding to the chunk's context — a section-4 chunk from a pension bylaw doesn't collide semantically with a section-4 chunk from a different bylaw.

services/chunking.py build_contextual_input()
07

Embed

Batched embedding call with input_type="search_document". Cohere batches at 96/call, OpenAI at 256/call. Returns float vectors persisted onto each Chunk row.

services/embedding.py embed_texts() Cohere embed-multilingual-v3.0 OpenAI text-embedding-3-large
08

Chunk rows + BM25 index

Each chunk is inserted with its vector. Then a Postgres full-text index is built: UPDATE chunks SET text_search = to_tsvector('simple', normalize_hebrew(text)). The normalization does light morphological stripping (prefixes, suffixes, sofit).

routes/ingest.py (lines 251–266) services/hebrew_text.py normalize_hebrew()
Note. The Hebrew BM25 zero-recall bug was fixed on the query side (see "Known gaps" — normalize_hebrew_to_tsquery()); indexing is unchanged.
09

Metadata classification (background)

Spawned as background task classify_document_by_id_bg(). Feeds the first ~4k characters (first 8 chunks) to Claude Haiku, which extracts structured metadata: AI-generated title, doc type, folder, summary, key dates, meeting/decision numbers. Result stored in doc_metadata JSON column.

routes/documents.py classify_one_document() Claude Haiku 4.5 → doc_metadata
10

Amendment extraction (background, chained after classification)

If classification succeeds, the amendment extractor runs. Claude reads the document plus a list of prior documents in the same tenant, and outputs structured amendment edits: target section, action, new text, effective date, confidence. High-confidence edits auto-apply by setting chunks.superseded_by_amendment_id. Lower-confidence edits are written with needs_review=true for human review.

services/amendment_extractor.py extract_amendments() Claude Haiku 4.5 → amendment graph

01

Conversation resolution

If a conversation_id is supplied, the existing Conversation is fetched; otherwise a fresh single-turn conversation is created. This is what makes multi-turn context possible.

routes/search.py search_pipeline()
02

Chat context loading

Prior Query rows in the conversation are loaded in order. Each becomes a (role, text) pair — "user" for the question, "assistant" for the answer. This history feeds both triage and query rewriting.

routes/search.py (lines 173–186)
03

Triage (clarify vs answer)

Haiku decides whether the question is answerable given the corpus and conversation context, or whether it needs clarification. If ambiguous — for example, referring to an entity never defined in the conversation — it returns mode="clarify" and the pipeline short-circuits.

services/chat_triage.py triage_turn() Claude Haiku 4.5
04

Query rewriting

Fuses conversation history and lexicon matches into a single canonical query. Example: user says "ירשתי בית" after prior discussion of the pension bylaw, and the rewriter produces "מה המשמעות של שיוך דירות אם אני יורש של חבר שנפטר". The rewritten query is used identically for both vector and BM25 retrieval paths.

services/query_rewriter.py Claude Haiku 4.5
05

HITL cache lookup

The query is embedded (input_type="search_query") and cosine-matched against the AuthoritativeAnswer table — human-curated Q&A pairs. If similarity clears the per-answer threshold, the cached answer is returned directly, bypassing retrieval and generation entirely.

Cohere embed-multilingual-v3.0 short-circuit possible
06

Hybrid retrieval

Two lanes run in parallel and are then fused:

  • Vector. Top 64 chunks by cosine similarity. Per-document cap of 3 prevents any single large doc from saturating the candidate pool.
  • BM25. Query normalized via normalize_hebrew(), then executed as plainto_tsquery('simple', :q) against the text_search tsvector. Top 20.
  • Fusion. Reciprocal-rank fusion merges the two rankings. Top ~12 candidates advance.
services/retrieval.py hybrid_retrieve()
07

Rerank

The top ~15 candidates are reordered by a cross-encoder for relevance to the specific question. A diversity filter then caps at 2 chunks per document in the final top-K, so no single document monopolizes context.

services/reranker.py rerank() Cohere rerank-v3.5
08

Amendment overlay

For each retrieved chunk with a section_ref, the amendment chain is fetched — only active amendments where needs_review=false. These are formatted as human-readable notes and passed to the LLM alongside the sources, so the answer reflects the current effective wording, not the pre-amendment text.

services/retrieval.py _amendment_chain_for_chunks()
09

LLM answer generation

Sonnet 4.6 is called with tool-use forced to an answer schema — {confidence, answer, references}. The prompt includes: system prompt (52-section Hebrew instructions on bylaw interpretation, source hierarchy, no-hallucination rules), last 8 conversation turns, lexicon block, amendment notes, and the retrieved chunks numbered [1]…[N].

services/llm.py answer_with_citations() Claude Sonnet 4.6 → structured JSON
10

Citation binding & query logging

The chunk IDs referenced in the answer are turned into SourceCitation objects (filename, section_path, text). A Query row is written to the DB: question, answer, source chunk IDs, confidence, LLM-used flag, and a retrieval_debug payload with per-lane scores (vector / BM25 / fusion / reranked). Near-miss authoritative answers are computed for UI display. Response returns as a SearchResponse with sources, references, and the debug payload.

→ SearchResponse → Query row (for observability)

Known gaps & watch-outs

Places where the pipeline is thinner than the code suggests. Track these against the golden set as the roadmap items land.

BM25 Hebrew normalization returned 0 hits

Real problem was on the query side, not indexing. plainto_tsquery AND'd every normalized form so any doc had to contain all alternatives at once. Fixed 2026-07-08 with normalize_hebrew_to_tsquery(): per-source-word OR alternation, cross-word AND, Hebrew stop-word list. Retrieval now uses to_tsquery. Index unchanged.

Retrieval is a black box to the answer step

The LLM sees only pre-retrieved chunks. It cannot ask for a broader search, a different collection, or the amendment chain of a specific section. This is what roadmap §1.3 (wrap as tools) and §1.4 (agentic loop) address.

Hierarchy is implicit

section_path is a flat string; collections and doc types live in metadata but are not first-class entities. Cross-document relationships (decision cites bylaw, protocol contains decisions) exist informally at best. Roadmap §1.2 makes hierarchy first-class.

No enforced cite-or-refuse guardrail

Landed 2026-07-09. Post-generation validator in llm._enforce_cite_or_refuse downgrades any confident answer to refused when references are empty or when no reference title matches a retrieved chunk's document filename. Prompt §3.5 also names the check explicitly so the LLM self-refuses more often. Refused UI reframed to signal integrity rather than failure, with a "the corpus should know this" flag straight into the super-admin debug queue.

No tenant context injection

Landed 2026-07-09. System prompt now composed as PREFIX + tenant.system_context + SUFFIX. Super-admin edits the middle block per tenant in the admin panel. אל-רום's original hierarchy + precision rules seeded into the DB on deploy. Empty tenants fall through to a generic template (just identity + recency/specificity hierarchy + no-fabrication rules) — no invented structure.

Triage relies on hand-tuned heuristics

Edge-case ambiguous queries can slip through triage and reach generation, where the LLM's own self-clarification converts them back to clarify-mode post-hoc. Golden-set expansion (roadmap cross-cutting) should surface these.
