Fix BM25 Hebrew normalization — per-word OR tsquery §1.1
Root cause: plainto_tsquery AND'd every normalized form so a doc had to contain all of them at once. 1
Foundation

Core engine — hierarchy, context, tooling

Focus. Give the system the same organizational scaffolding and reasoning surface a smart human researcher would need. Move away from embedding-first RAG as the sole retrieval path; treat retrieval as a tool surface the LLM can call iteratively.

1.1 Fix the retrieval primitive

  • Real Hebrew morphology in normalize_hebrew — prefix (ה / ו / ב / כ / ל / מ / ש) and inflection handling.
  • Section-path and folder filters exposed to the retrieval layer.
  • Folder / collection metadata pushed into the chunk index for scoped search.
  • Chunking strategy revisit — semantic vs fixed, overlap, section-aware for bylaws.
  • Reranker layer confirmed end-to-end against the golden set.
Benefit. Actual BM25 recall on inflected Hebrew queries (today's silent 0-hit path starts working). Scoped search becomes possible. Everything downstream inherits the improvement.

1.2 Hierarchy construction & document structure

Building a rich, machine-navigable hierarchy over the corpus is what turns retrieval from keyword-lookup into structured reasoning. Same principle a smart researcher would apply — "where would this live?" before "what does it say?"

  1. Explicit hierarchy model: tenant → collection → doc_type → document → section → subsection. Persisted as first-class entities with parent/child relationships, not implicit in filenames.
  2. Automatic hierarchy inference — LLM-assisted classification of newly ingested documents into the right collection + doc_type, with confidence scores and review queue for low-confidence placements.
  3. Section-tree extraction — parse structural markers (סעיף, פרק, החלטה, סעיף משנה, נספח) into a proper tree, not a flat section-path string. Enables ancestor/descendant queries.
  4. Cross-document relationships — a decision "cites" a bylaw section; a protocol "contains" decisions; an amendment "modifies" a section. First-class graph edges, queryable and displayable.
  5. Per-tenant hierarchy customization — tenants have different governance structures (committees, sub-organizations); the hierarchy adapts rather than forcing a one-size-fits-all tree.
  6. Hierarchy-aware chunking — chunks carry their full ancestor path (collection → doc → section → subsection) as metadata, retrievable and filterable.
  7. Hierarchy browse tool for the agent — get_hierarchy(path?, depth?) so the agent can zoom in and out ("what's in the pension collection?", "what sections does bylaw X have?") before drilling into content.
  8. Human-editable hierarchy — admins can restructure, rename, merge, or split collections without breaking retrieval or amendment links.
  9. Hierarchy visualization — a browseable tree view of the corpus for humans (feeds into §3.1 faceted browse) and a compact serialized form for the LLM context.
Benefit. The single biggest lever for agentic retrieval quality. An LLM that can reason "this question is about a decision, so search the decisions collection first, in the pension committee, from the last 3 years" is dramatically more accurate than one blindly searching all chunks. Also unlocks per-type UX in §3.3 and per-collection ACLs in §3.2.

1.3 Wrap retrieval, amendments & HITL as tools

  • search_documents(query, folder?, doc_type?, top_k)
  • get_document_toc(doc_id)
  • read_section(doc_id, section_path) — current effective version.
  • list_amendments(doc_id, section?)
  • check_authoritative_answers(question) — HITL cache probe.
Benefit. Stable retrieval API reused by RAG, agent, mailer, integrations. Preserves the amendment graph and HITL cache. Retrieval quality becomes independently testable against the golden set.

1.4 Agentic loop on top of the tools

  • Replace one-shot "retrieve → answer" with a Sonnet loop that iterates tool calls.
  • Agent decides when to search, drill in, check amendments, consult HITL.
  • Every tool call logged for traceable debugging.
Benefit. Multi-hop questions become tractable — bylaw → section → amendment → answer. Self-correction on empty retrievals. Cache-first via check_authoritative_answers.

1.5 Tenant context injection

  • Per-tenant system-prompt block: who they are, governance structure, main folders, glossary, common query patterns.
  • Query rewriting / HyDE retained where useful — no longer carrying the retrieval burden alone.
Benefit. Cheapest single quality lever. Correct terminology per organization. Feeds tool-call reasoning ("search this folder first"). First step toward multi-tenant differentiation.

Trust & safety (folded in)

  • Hallucination guardrails.
  • "Cite or refuse" mode.

Cross-cutting: golden-set evaluation

  • Expand golden-set coverage: single-hop lookup, multi-hop amendment trace, ambiguous / off-topic (guardrail tests).
  • Binary pass/fail per question; re-run after every change.

Deprioritized: A/B testing (too much overhead for team size); recall as headline metric (vanity).

Go-live

User experience

Focus. Is it good enough for secretaries and managers on day one? Four surfaces get the day-one lift — support, onboarding, chat, and documents. Talia leads across all four; each has a concrete P0/P1 slice ahead of go-live.

2.1 Support experience

A secretary who's stuck at 22:00 needs help without a phone call to Tal. Support has to feel present without becoming a distraction on the main surface.

  1. In-app support widget — chat channel (Intercom / Crisp / equivalent) with session context (tenant, role, current page, last query) attached automatically. Bottom-corner, dismissible.
  2. Contextual help — inline "?" tooltips next to non-obvious features (amendment review, HITL cache, folder filters, "why refused" explainer).
  3. One-click bug / feedback report — grabs current URL, recent actions, optional screenshot; routes into the debug queue with a "user-reported" tag.
  4. Self-serve diagnostics — "why did I get no answer?" surface exposing the retrieval trace in a friendly form (not raw JSON).
  5. Public status page for platform-wide incidents.
  6. Ticket lifecycle visible to the reporting user — "your issue is being worked on" beats silence.
Benefit. Every ticket that gets resolved without a human touch is a step toward operating at more than a handful of tenants. Also feeds §7 sales — support quality is a churn signal.

2.2 Onboarding experience

Formalize the path from first sign-in to daily use. First-run polish should get every new user to first value in under two minutes, without a call.

  1. Welcome email on account creation — tenant-branded, one clear CTA to the first action per role.
  2. Role-aware landing — secretary lands on "upload your first document"; end user lands on Q&A with 3–5 suggested-question chips from tenant context.
  3. Empty states across every surface — no "no data" screens; each empty view teaches the next action.
  4. Progressive disclosure — advanced surfaces (bulk ops, amendment review, HITL cache) unlock as usage grows, not on day one.
  5. Sample corpus / demo tenant for evaluators and sales calls — no more staring at an empty box during a pitch.
  6. Tenant-admin onboarding runbook — a separate flow for the person setting up the tenant vs the end users.
  7. Onboarding checklist (later) — visible until dismissed: upload first doc, try a query, invite a colleague, set notifications.
Benefit. Turns onboarding from bespoke Tal-time into a repeatable, measurable path. Precondition for scaling beyond design-partner customers. Metrics feed §5 (time-to-first-query, D7 return).

2.3 Chat enhancements

The Q&A surface is where 90% of user attention lives. Small polish here compounds.

  1. Clearer confidence states — "confident + cited" vs "low-confidence guess" vs "refused" render as three visually distinct answer types, not one style with a badge.
  2. Refined refused state — integrity framing (already shipped in §1) reinforced with a one-click "the corpus should know this" path into the debug queue.
  3. Suggested next questions — after every answer, 2–3 follow-ups derived from the retrieved chunks. "Did you also want to see…"
  4. Recent-searches surface — quick access to the last 10 queries per user; pinnable.
  5. Inline citation preview — hover a citation → mini-preview of the surrounding paragraph without opening the PDF.
  6. Copy answer with citations — one button that copies the answer + inline source references, ready to paste into an email.
  7. Query rewriting hints — when the retrieval was empty, suggest a rewrite ("try mentioning the committee name") rather than a bare refusal.
  8. Conversation memory polish — visible "this thread" indicator when a follow-up builds on the previous turn.
  9. Lexicon-term highlighting in the chat input — as the user types, any word matching an active lexicon entry renders in an accent color with a subtle underline. Signals "we know this term and will expand it in retrieval" and turns the invisible query-rewriting step into visible trust. Hover reveals the expansion.
Benefit. Increases both perceived quality and actual trust per answer. Follow-up suggestions and refused-to-flag path directly reduce zero-result frustration, the top complaint in early usage.

2.4 Document enhancements

Documents are the trust artifact. Every improvement here reinforces "the answer is grounded in something real you can open."

  1. Jump-to-page from citations — click a citation → PDF opens at the referenced page, not just the file (v1 opens the file; this is the v2 upgrade).
  2. Passage highlight on open — the cited paragraph is visually marked in the PDF when the viewer opens.
  3. Rich document detail page — beyond the current file view: metadata sidebar (type, date, uploader), amendment chain, related documents, "questions this document answered."
  4. Doc-type badges throughout the UI — bylaws / decisions / protocols / contracts each get a distinct color + icon so the user always knows what kind of source they're looking at.
  5. In-viewer search — Ctrl+F inside a document, RTL-aware, works on OCR text layer too.
  6. Side-by-side mode — original PDF on one side, extracted / answered text on the other. Useful for reviewers.
  7. Print / download the original with tenant-level ACL respected.
  8. Better cards on the Documents browse view — richer previews, better visual hierarchy, faster scan.
Benefit. Closes the trust loop end-to-end: retrieved → cited → opened at the exact spot → verified. This is the single biggest lever for "the platform feels serious" perception on the first visit.

Cross-cutting

  • Mobile polish — every above surface has to survive a secretary opening it on a phone in a meeting. Not a separate track, a constraint on every §2 item.
  • Design system — Talia owns the shared visual language across Takanon and Klaser Meetings so §2 improvements ship consistent, not one-off.

Client-signaled

Document management & archival integration

Focus. The platform is the tenant's system of record for governance documents — not just a Q&A surface over uploads. Secretaries want to sift, browse, and search their corpus, and to integrate with the archives they already run.

3.1 Faceted browse & structured search

  1. Corpus browser as a first-class surface (not just Q&A). List / grid view of all documents.
  2. Filter sidebar over doc_metadata: doc type, folder, effective date range, status (active / amended / repealed), authority, tags, uploader.
  3. Sort by date, title, last modified, relevance.
  4. Structured search: title contains, decision number = X, meeting number range, effective between dates — combinable with full-text search inside the current filter.
  5. Saved views ("all active pension bylaws", "unclassified uploads this month").
  6. Bookmarks / pinned documents per user.
Benefit. Matches how secretaries actually work — they usually know roughly what they're looking for and want to narrow down. Turns the platform from Q&A box into daily working environment.

3.2 Repository structure

  1. Folder / collection hierarchy (bylaws vs decisions vs protocols vs contracts).
  2. Rich metadata surfaced as filter dimensions in 3.1.
  3. Amendment / supersession graph (extend existing work).
  4. Version history and diff view.
  5. Bulk operations: upload, retag, move, reclassify, delete.
  6. Access control per collection.

3.3 Doc-type expansion (first-class governance artifact types)

  1. Per-type schemas — each artifact type gets structured metadata beyond generic tags:
    • Bylaws — sections, effective ranges, superseded-by chain.
    • Decisions — committee, decision number, date, outcome, cites bylaw sections.
    • Protocols (minutes) — meeting date, committee, attendees, agenda items, contained decisions, action items.
    • Contracts — parties, effective / expiry dates, obligations, renewal terms.
    • Correspondence, financial reports, permits — added as needed.
  2. Per-type browse UX — decision registry view, protocol calendar view, contract expiry dashboard.
  3. Per-type semantics in the amendment graph — a decision doesn't amend a bylaw the way an amendment does.
  4. Per-type extraction prompts extending the classifier.
  5. Per-type retrieval filters exposed to the agent (§1.2 tools).
Benefit. Unlocks queries the current pipeline can't answer well — "list all pension-committee decisions from 2024 that reference section 4 of the housing bylaw." Turns generic RAG into a governance-specific system.

3.4 Archival system integration

  1. Google Drive / SharePoint import — preserve source folder structure, filenames, and existing metadata.
  2. Bulk archive import: point at a Drive tree or an uploaded zip and ingest wholesale.
  3. One-way sync (external → El-Rom) as v1; round-trip sync explicitly held as a v2 decision.
  4. Deduplication: same bylaw arriving from multiple sources — detect and merge.
  5. Provenance & attribution: source path, uploader, ingestion timestamp, channel — visible in the doc detail view.
  6. Local filesystem / Windows share import for tenants without cloud archives.
Benefit. Lowers switching cost — tenants extend existing archives rather than abandoning them. Direct answer to the "integrate with our archival system" ask.

3.5 Original document viewer

Today the platform displays a rendered/extracted view of the document. Users need to open the original file (PDF, DOCX, scanned image) in a proper web viewer embedded in the platform — not download-and-open in an external app, and not our re-rendering.

  1. In-app web viewer for PDF and DOCX — page navigation, zoom, search within the document.
  2. Click-through from any citation, chunk, or search result opens the original at the correct page / section.
  3. Deep-link support — a URL points to a specific document page or section for sharing and bookmarking.
  4. Text-layer overlay on scanned PDFs — OCR text stays selectable and searchable in the viewer so citations remain jump-to-source even on image-based scans.
  5. Highlight the cited passage on open — the section the LLM referenced is visually marked in the original.
  6. Side-by-side mode — original document on one side, extracted/answered text or metadata on the other.
  7. Print / download the original file, with tenant-level ACL enforcement.
  8. Viewer works on mobile — pinch-zoom, page swipe, readable on small screens.
  9. Format coverage roadmap — PDF and DOCX in v1; images (JPG/PNG), spreadsheets, and legacy DOC as fastfollows.
Benefit. Trust and verifiability. Secretaries and reviewers need to see the actual signed / scanned document, not our interpretation of it — especially for bylaws and decisions where the exact wording is legally meaningful. Also closes a common regression path where extraction loses formatting, tables, or signatures that matter.

  2. In-app support widget — chat / message channel (Intercom, Crisp, or equivalent) for direct user-to-support communication. Attaches session context (tenant, role, current page) automatically.
  3. Bug report & feedback capture — one-click surface that grabs the current URL, recent actions, and (optional) screenshot; routes to the internal tracker.

New product · Klaser brand

Meeting capture & minute-taking

Focus. A separate product under the Klaser brand, alongside Takanon. Upload meeting audio → transcript → structured draft protocol. Owned end-to-end by Gil. Feeds Takanon via the shared decision registry, but stands alone commercially.

4.1 Audio → draft protocol

  1. Upload meeting audio (or video) — Hebrew-tuned transcription, RTL-aware.
  2. Speaker diarization; map speakers to known attendees where possible.
  3. Structured extraction: attendees, agenda items, decisions, action items with owners.
  4. Auto-generated draft protocol (per tenant template) queued for secretary review.
  5. Decision entries auto-created and linked to the source protocol; classified into the doc-type registry from §3.3.
  6. Review UI — secretary approves / edits before the protocol becomes canonical.
Benefit. Compresses the "meeting → typed protocol" workflow from days to minutes. Solves the highest-friction task in the secretary's week.

4.2 Governance workflow

  1. Send draft protocol to attendees for approval (integrates with §6 Mailer).
  2. Attendee comments / correction requests tracked against the draft.
  3. Approved protocol locks; decisions promote to canonical registry.
  4. Action-item tracker — items surface in the mailer digest until closed.
Benefit. Closes the loop: meeting recorded → protocol drafted → decisions logged → participants notified → action items tracked.

Sales CRM

Focus. Internal tool for tracking the Klaser sales pipeline — not a customer product. Prospect universe in the hundreds (every kibbutz and moshav in Israel is addressable), so a real system beats scattered spreadsheets and WhatsApps from day one. Two-week delivery target. 7.0 Buy vs build — decide first

Open question, answered before any code. The right answer isn't obvious at hundreds-of-prospects scale.

Off-the-shelf candidates

  • HubSpot Free / Starter — familiar UX, generous free tier, weak on custom lifecycle sync.
  • Pipedrive — pipeline-first, cheap per seat.
  • Attio — modern, API-first, strong for custom objects (would map cleanly to tenant records).
  • Airtable / Notion — cheapest, most flexible, weakest as a sales surface for daily use.

Decision criteria

  • Cost at hundreds of contacts, single-digit seats.
  • API surface for the tenant-lifecycle sync (§7.2) — critical.
  • Hebrew UX / RTL for anyone else logging notes.
  • Time-to-live vs. the two-week build target.
  • Exit cost — how bad is it to move off later?
Bias. Start with Attio or HubSpot Free and only build if a real integration or workflow limit hits. Buying wins on the first two weeks; building only wins if the CRM has to become the operational hub of the business (tenant provisioning, billing, health signals).

Foundational · Shared platform

Shared infrastructure — Klaser identity service

Focus. Klaser is a multi-product brand — Takanon today, Klaser Meetings next, more later. This track builds the shared identity, tenants, and subscriptions layer that both products (and every future product) sit on top of. Full plan in klaser-platform-infra.md. Decision: Approach B1 — a dedicated identity service, own repo, own Render project, own Postgres. No shortcuts.

9.1 The three-service shape

Three independent services replacing the current single-service layout:

  1. klaser-identity (new) — users, tenants, sessions, subscriptions, Google OAuth callback, email/password, invites, resets. Own DB. Hosted at auth.klaser.co.il.
  2. klaser-takanon (rename of elrom-platform) — docs, chunks, lexicon, amendments, goldens. No user or tenant tables anymore. Frontend at takanon.klaser.co.il.
  3. klaser-meetings (new, Gil owns) — recordings, transcripts, protocols. Same auth pattern. Frontend at meetings.klaser.co.il.
Benefit. Every product deploys on its own cadence. A third product plugs in without a migration. אל-רום appears as the same tenant in all products because tenants live in one place.

9.2 Auth flow

  1. User hits a product frontend → cookie check on .klaser.co.il.
  2. If no session → redirect to auth.klaser.co.il/login?redirect=….
  3. Identity handles Google OAuth (single client) or email+password, sets cookie on .klaser.co.il, redirects back.
  4. Product backend calls identity /api/me with the cookie → gets { user, tenant, entitlements: [...] }.
  5. Backend checks its entitlement is in the list, or 403s.
Benefit. Sign in once, land in any product. Cookie-scope trick avoids building an auth proxy.

9.3 Data ownership rule

  • Identity DB: users, tenants, user_tenants, subscriptions, auth_tokens, sessions. Nothing else.
  • Takanon DB: everything doc-related. User/tenant IDs are UUIDs — no cross-DB joins.
  • Meetings DB: everything meeting-related. Same rule.
  • Enrichment across DBs is always an HTTP call to the identity API, never a direct DB read.

9.4 Product switcher & entitlements

  1. Each frontend renders a header switcher from me.entitlements.
  2. ["takanon"] → no switcher; ["takanon", "meetings"] → both shown, deep-link to the other.
  3. Entitlements live in the identity DB (subscriptions table: tenant_id, product, plan, active, expires_at).
  4. Product IDs are stable strings — takanon, meetings. Adding a product = adding an entitlement, not a schema change.

9.5 Service-to-service auth

  1. Each product backend gets a long-lived service token for the identity API.
  2. Used for background jobs, cron, admin scripts — anything outside a request context.
  3. Passed as Authorization: Bearer <service-token>. Per-product, rotatable.

9.6 Migration order

  1. Stand up klaser-identity — port auth routes, users/tenants tables from Takanon backend. Same schema, new home. ~1 week Tal.
  2. One-shot data migration — dump users + tenants + auth_tokens from Takanon DB into identity DB.
  3. Point Takanon backend at identity — swap local auth code for HTTP calls; keep the same session cookie so nobody gets logged out.
  4. Widen cookie scope to .klaser.co.il.
  5. Gil starts klaser-meetings against the live identity service. No temporary local auth — ever.
Rollback. Every step above is reversible until step 2 completes. Once data is migrated, identity DB is authoritative.

9.7 Render layout

  • 3 Render projects (one per service).
  • 3 web services (backends) + 3 static-site services (frontends).
  • 3 managed Postgres instances (identity can be smallest tier).
  • All under klaser.co.il so the parent-domain cookie works.
  • Cost delta vs "identity inside Takanon" — ~$20–30/month. Cheap vs a future migration.

Non-negotiables

  • Do not build a second users table in either product.
  • Do not create a second Google OAuth client.
  • Do not scope the session cookie to a specific subdomain — must be .klaser.co.il.
  • Do not have Meetings call Takanon's backend for auth (that's the abandoned B2 shape).
  • Do not put shared code in cross-repo shared/ folders and expect it to work; publish a package or copy-paste until it stabilizes.

01

Governance operating system

Extend upstream from bylaws and decisions to the full lifecycle: meeting agenda → discussion → vote → decision → automatically becomes searchable, versioned bylaw. Become the system of record for governance, not just a search layer on top.

Member-facing side (B2B2C)

Open a resident-facing surface — "what are my rights re: parking?", "what did the assembly decide about the pool?" Turns the platform from admin tool into community infrastructure.

Cross-community precedent network

Anonymized decision/bylaw corpus across kibbutzim & moshavim, so a secretary can ask "how did others handle a member leaving with equity?" Every new customer improves the product for all — data network effect as the moat.

Regulatory / legal layer

Proactive advisor against רשם האגודות השיתופיות rules and national regulation. "Your bylaw conflicts with new regulation X"; "decision from 2019 needs review."
